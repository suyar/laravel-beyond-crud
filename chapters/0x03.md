# 动作（Actions）

现在我们可以以一种类型安全和透明的方式来处理数据，我们需要开始用它做一些事情。

就像我们不希望使用充满随机数据的数组一样，我们也不希望我们项目中最关键的部分，即业务功能，被分散在随机的函数和类中。

这里有一个例子：你项目中的一个场景（stories）可能是『管理员创建发票（an admin to create an invoice）』。这意味着我们将在数据库中保存一张发票，但要做的事情不止这些。

单独计算发票每个项目的价格和总价格

- 生成发票编号（Generate an invoice number）
- 保存发票信息到数据库（Save the invoice to the database）
- 通过支付提供者创建一笔支付（Create a payment via the payment provider）
- 创建包含所有相关信息的 PDF（Create a PDF with all relevant information）
- 将 PDF 发送给客户（Send this PDF to the customer）

在 Laravel 中一个常见的做法是创建『胖模型（fat models）』来处理所有这些功能。在这一章中, 我们将探讨另一种方法来将这种行为添加到我们的代码库中。

我们将把这些用户故事（user stories）作为项目的一等公民，而不是把功能混入模型（models）或控制器（controllers）中。我倾向于将这些称为『动作（actions）』。

## 术语（Terminology）

在研究它们如何使用之前，我们需要讨论动作（actions）是如何构成的。首先，它们存在领域层中。

其次，它们是没有任何抽象或接口的简单类。一个 `Action` 是一个接受输入，做一些事情，并给出输出的类。这就是为什么一个 `Action` 通常只有一个公开方法，有时还包含一个构造函数。

作为我们项目中的一个惯例，我们决定对所有的类添加后缀。当然，`CreateInvoice` 听起来不错，但当你要处理几百或几千个类时，你要确保不会发生命名冲突。你看，`CreateInvoice` 很可能也是一个可调用的控制器（invokable controller）、一个命令（command）、一个任务（job）或一个请求（request）的名字。我们希望尽可能地消除混淆，因此，`CreateInvoiceAction` 是比较好的名字。

显然，这意味着类的名称变得更长。实际情况是，如果你在做较大的项目，你无法避免使用较长的名字，以确保没有混淆的可能。下面是我们一个项目中的极端例子（我没有开玩笑）：`CreateOrUpdateHabitantContractUnitPackageAction`

我们一开始很讨厌这个名字。我们拼命地想出一个更短的名字。但最后，我们不得不承认，明确一个类的内容是最重要的事情。反正我们的 IDE 的自动补全功能会弱化长名字带来的不便。

当我们确定了类的名称后，要克服的下一个障碍是为使用我们的动作（actions）的公共方法命名。一种选择是让它成为可调用的（invokable），就像这样：

```
class CreateInvoiceAction
{
    public function __invoke(InvoiceData $invoiceData): Invoice
    {
        // ...
    }
}
```

但这种方法实际上有一个问题。为了理解这个问题，我需要提到一些我们在后面章节会看到的东西，那就是有时候，我们会组合使用其他动作（actions）。它看起来就像这样：

```
class CreateInvoiceAction
{
    private CreateInvoiceLineAction $createInvoiceLineAction;

    public function __construct(
        CreateInvoiceLineAction $createInvoiceLineAction
    ) { /* ... */ }

    public function __invoke(InvoiceData $invoiceData): Invoice
    {
        foreach ($invoiceData->lines as $lineData) {
            $invoice->addLine(
                 ($this->createInvoiceLineAction)($lineData)
            );
        }
    }
}
```

看到问题了吗？PHP 不允许直接调用一个类上的可调用（invokable property）的属性，因为它会寻找一个类的方法（method）。这就是为什么你必须在调用该 `Action` 之前用括号把它包起来。

如果你不想使用这种古怪的语法, 还有其他的选择: 例如, 我们可以使用 `handle`，Laravel 经常在这种情况下使用它作为默认名称。这又是一个问题，特别是 Laravel 使用它。

每当 Laravel 允许你使用句柄，例如，作业（jobs）或命令（commands），它也会提供依赖注入。在我们的动作（actions）中，我们只希望构造函数有依赖注入的功能。我们将在本章的后面仔细研究这背后的原因。

所以 `handle` 也被淘汰了。当我们开始使用动作（actions）时，我们实际上对这个命名难题进行了很多思考。最后我们决定采用 `execute`。但请记住，你可以自由地想出你自己的命名规则，因为这里的重点是关于使用动作（actions）的模式，而不是它们怎么命名。


# 实践（Into practice）

抛开所有的术语，让我们谈谈动作（actions）有啥用，以及如何在实际场景中使用。

首先让我们来谈谈可复用性。使用动作（actions）的诀窍是把它们分割成足够小的部分，以便代码复用，同时保持它们足够大，以免类过多。以我们的发票为例：从从发票生成一个 PDF 可能会出现在我们应用里的多种场景。当然，在实际创建发票时，会生成 PDF，但管理员也可能在发送前想看到预览或草稿。

这两个用户故事点（user stories）：『创建发票』和『预览发票』显然需要两个入口点，因此需要两个控制器（controllers）。但另一方面，根据发票生成 PDF 在这两种情况下都需要去实现。

当你开始花时间考虑应用程序究竟要做什么时，你会注意到有很多动作（actions）可以被复用。当然，我们也需要注意不要过度抽象我们的代码。复制粘贴一点代码往往比过早地进行抽象要好。

进行抽象时，一个好的经验法则是考虑功能点而不是代码的技术点。当两个动作（actions）可能做类似的事情，但在两个完全不同的上下文中使用时，你不应该过早开始抽象它们。

另一方面，有些情况下，抽象是有用的。再以我们的大票生成 PDF 为例：你有可能需要在其他地方也生成 PDF，而不仅仅是发票 —— 至少在我们的项目中是这样的。使用一个通用的 `GeneratePdfAction` 接口（interface），提供一个 `toPDF` 方法，然后 `Invoice` 实现这个接口（interface）。这意味着生成 PDF 的功能现在可以在整个项目中重复使用。

但是说实话，我们的大部分动作（actions）都是针对用户故事（user stories）的，而且不会被重复使用。你可能会认为，在这些情况下，动作（actions）是不必要的开销。但请坚持住，因为可复用性并不是使用它们的唯一原因。实际上，最重要的原因与技术收益完全无关：动作（actions）使得程序员以更接近真实世界的方式来思考，而不是以代码的方式。

假设你更改发票的创建方式。一个典型的 Laravel 应用可能会把发票创建逻辑分布在一个控制器（controller）和一个模型（model），也许还有一个生成 PDF 的作业（job），最后使用事件监听器（event listener）来发送发票邮件。你需要知道这些都放在哪些地方。我们的代码又回到了原先分散在代码库中的情况，按技术属性分组，而不是按其意义分组。

动作（actions）减少了这种系统所带来的认知负担。如果你需要研究如何创建发票，你可以简单地从动作类（action class）开始。

不要误会：动作（actions）可以很好地与例如异步作业（asynchronous jobs）和事件监听器（event listeners）一起工作；尽管这些作业（jobs）和监听器（listeners）只是作为动作（actions）的基础设施，而不是业务逻辑本身。这是为什么我们需要区分领域层（domain）和应用层（application）的一个很好的例子：每个层都有自己要做的事情。

现在，我们现在既有了复用性，又减少了认知负荷，但不止这些！

由于动作（actions）是几乎独立存在的小快代码，所以很容易对它们进行单元测试。在测试中，你不需要担心模拟 HTTP 请求（fake HTTP requests），创建门面（making facades）等等。你可以简单地创建一个新的动作（action），也许需要提供一些模拟的依赖关系，将所需的输入数据传递给它，并对其输出进行断言。

例如，让我们以 `CreateInvoiceLineAction` 为例。它将：

- 获取将被开票的产品数据（take data about which product will be invoiced）
- 接收金额 + 期限（receive an amount plus a period）
- 计算总价（calculate the total price and then）
- 计算含税和不含税的价格（calculate prices with and without VAT）

你可以为这些编写简单又强大的单元测试。

如果你的所有动作（actions）都经过适当的单元测试，你可以非常确信应用程序提供的大部分功能都可以按预期工作。现在只需根据用户的使用方式调用这些动作（actions），并为这些部分编写一些集成测试。

## 组合动作（Composing actions）

我之前已经简单地提到过动作（actions）的一个重要特征，就是它们如何使用依赖注入。我们利用容器在构造函数中注入数据，使用 `execute` 方法传入上下文数据，所以我们可以自由地组合动作与动作（actions out of actions）...

你可能会有些想法。我知道，你想要避免深层次的依赖 —— 它使代码变得复杂，并高度依赖对方 —— 然而在一些情况下，使用 DI 是非常棒的。

再以 `CreateInvoiceLineAction` 为例，它需要计算增值税。现在，根据上下文，一个发票项目（invoice line）可能有包含或不包含增值税。计算增值税是件小事，我们不希望 `CreateInvoiceLineAction` 关心它的细节。

假设我们有一个简单的 `VatCalculator` 类 —— 这是可能是在 `\Support` 命名空间的东西 —— 它可以这样被注入：

```
class CreateInvoiceLineAction
{
    private VatCalculator $vatCalculator;

    public function __construct(VatCalculator $vatCalculator)
    {
        $this->vatCalculator = $vatCalculator;
    }

    public function execute(
        InvoiceLineData $invoiceLineData
    ): InvoiceLine {
        // ...
    }
}
```

你大概是这样来使用：

```
public function execute(
    InvoiceLineData $invoiceLineData
): InvoiceLine {
    $item = $invoiceLineData->item;

    if ($item->vatIncluded()) {
        [$priceIncVat, $priceExclVat] =
        $this->vatCalculator->vatIncluded(
            $item->getPrice(),
            $item->getVatPercentage()
        );
    } else {
        [$priceIncVat, $priceExclVat] =
        $this->vatCalculator->vatExcluded(
            $item->getPrice(),
            $item->getVatPercentage()
        );
    }

    $amount = $invoiceLineData->item_amount;

    $invoiceLine = new InvoiceLine([
        'item_price' => $item->getPrice(),
        'total_price' => $amount * $priceIncVat,
        'total_price_excluding_vat' => $amount * $priceExclVat,
    ]);
}
```

`CreateInvoiceLineAction` 被注入到 `CreateInvoiceAction`。还会有其他提来想，例如：`CreatePdfAction` 和 `SendMailAction`。

正如你看到的，灵活组合可以帮助你保持单个动作（action）的最小化，又能对复杂的业务功能以清晰和可维护的方式进行编码。

## 动作的替代方案（Alternatives to actions）

这里我要点一下，有两种设计模式不需要动作（actions）这个概念。

第一种是熟悉 DDD 的人都会知道的：`commands` 和 `handlers`。动作（actions）就想它们的简化版。`commands` 和 `handlers` 区分了需要发生什么以及如何发生，动作（actions）将这两种职责合二为一。当然，command bus 确实比动作提供了更多的灵活性。但是另一方面，它也需要你写更多的代码。

对我们项目的规模来说，将动作（actions）分割成 `commands` 和 `handlers` 太繁琐了。我们不需要为了额外的灵活性而花费大量的时间去写一堆代码。

第二种方案是值得一提的事件驱动系统（event driven systems）。如果你曾经使用过事件驱动系统，你可能会认为在需要的地方直接调用动作（actions）会过于耦合。还是那句话：事件驱动系统提供了更多的灵活性，然而对于我们的项目来说，使用它们就显得有些多余了。此外，事件驱动系统增加了一层间接层，使得代码更加复杂。虽然这种间接层确实有好处，但对我们来说，它将增加我们的维护成本。

我希望明确一点，我并不是说我们已经想好了一切，并且拥有所有 Laravel 项目的完美解决方案。我们并没有。当你继续阅读这本书的时候, 你一定要注意你的项目的具体需求。虽然你可能会使用这里提出的一些概念，但你也可能需要一些其他的解决方案来解决某些方面的问题。

对我们来说，动作（actions）是正确的选择，因为它们提供了适量的灵活性、复用性，并大大降低了认知负荷。它封装了应用程序的精髓（essence）。事实上，它与 DTO、模型（models）一起，可以被认为是项目的真正核心。

这就让我们进入了下一章，核心的最后一部分：模型（models）。
