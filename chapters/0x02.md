# 数据的使用（Working with data）

我使用领域对代码进行分组的思想 —— 我在上一章中解释过 —— 作为我们用来构建的基本理念。你会注意到在本书的前面部分中，这个核心思想会反复地出现。


早期，我们将以此为基础，再次展示出它的简单与强大：我们将使用结构化的方式对其建模；我们将使数据成为我们编码的一等公民。

在每个项目的开始，你几乎都会注意到数据建模的重要性：你一般不会从控制器（controllers）和作业（jobs）开始写代码，而是从 Laravel 的模型（models）开始。在大型项目中制作 `ERD` 和其他类型的图标，可以协助概念化应用程序要处理的数据。只有这些明确了，你才可以构建使用数据的入口（entry points）和钩子（hooks）。

本章的目的是告诉你数据流的重要性。我们甚至都不去讨论模型（models）。相反，我们将从单纯的数据入手，目的是让你团队中的所有开发人员都能写出可预测、安全且能与这些数据交互的代码。


为了能真正体会到应用一个简单的面向数据的模式所带来的所有好处，我们需要先深入了解一下 PHP 的类型系统。

## 类型理论（Type theory）

不是所有人对类型系统的术语理解都一样。因此，我们先明确几个术语。


类型的强度 —— 强类型或弱类型 —— 定义了一个变量在被定义后是否可以改变其类型。一个简单的例子：给定一个字符串变量 `$a = 'test'`；一个弱类型系统允许你将该变量重新分配到另一个类型，例如 `$a = 1`，变成一个整数。

PHP 是一种弱类型的语言。让我们来看看这在实践中意味着什么。

```
$id = '1'; // 例如，id 是从 url 参数获取的

function find(int $id): Model
{
    // 传进来的字符串 '1' 将被自动转换为 int 类型
}
find($id);
```

说白了：PHP 属于弱类型系统也是有道理的。作为一种主要使用 HTTP 请求的语言，所有东西基本上都是一个字符串。

你可能认为在高版本的 PHP 中，通过使用严格类型模式（strict types），可以避免这种隐式的类型转换，但这并不完全正确。

声明严格类型可以防止将其他类型传递给函数，但您仍然可以在函数本身中更改变量的值类型。

```
declare(strict_types=1);

function find(int $id): Model
{
    $id = '' . $id;

    /*
    * 这在 PHP 中是允许的
    * 现在， `$id` 变成了字符串
    */

    // ...
}

find('1'); // 会抛出错误

find(1); // 没问题
```

即使有严格的类型和类型提示，PHP 的类型系统也是弱鸡。类型提示只能确保一个变量在该时间点的类型，而不能保证该变量未来可能被赋予的任何值。

正如我之前所说的，PHP 的弱类型是有道理的，因为它处理的输入都是字符串。当然，强类型有它自己的特性：它带有一些保证。如果一个变量不能被改变类型，那么一系列未预期的行为就不会发生。

你看，从数学上可以证明，如果一个强类型的程序可以编译，那么这个程序就不会有一系列在弱类型语言中可能存在的错误。换句话说，强类型为程序员提供了更好的保证，即代码能按照预期的方式运行。

补充一下：这并不意味着强类型语言不会有错误！你也可以写出有 BUG 的代码。但是当一个强类型程序编译成功时，你可以确定该程序中不会出现与类型相关的错误和错误。

> 强类型系统可以让开发者在写代码的时候提供更多的审查，而不是在运行时才检查。

我们还需要关注一个概念：静态类型和动态类型系统，这就是事情开始变得有趣的地方。

你可能已经知道，PHP 是一种解释型语言，这意味着 PHP 脚本在运行时被翻译成机器代码。当你向运行 PHP 的服务器发送请求时，它将查找那些普通的 `.php` 文件，并将这些文件的内容解析成处理器可以执行的内容。

同样，这也是PHP的优势之一：写一个脚本，刷新页面，所见即所得。与那些必须先编译才能运行的语言相比，这是一个很大的区别。

很明显，有一些缓存机制可以优化这一点，所以上面的描述过于简单，但这足以让我们进入下一个问题。

这个问题就是，PHP 有一个缺点：因为它只在运行时检查其类型，所以可能会在运行时出现类型错误，使程序崩溃。你可能带着一个明确的错误去调试，但程序还是崩了。

这种运行时的类型检查使 PHP 成为一种动态类型语言。另一方面，静态类型语言会在代码执行前完成所有的类型检查，通常是在编译时。

从 `PHP 7.0` 开始，它的类型系统得到了很大的改进。以至于像 `PHPStan`，`Phan` 和 `Psalm` 这样的工具变得非常流行。这些工具采用了 PHP 的动态语言，但对代码进行了一系列的静态分析。

这些可选库不需要运行或者单元测试，也可以提供许多代码审查。像 PHPStorm 这种 IDE 也提供一些内置的静态检查。

了解了这么多背景知识以后，是时候回到我们应用的核心：数据（data）

## 构建非结构化的数据（Structuring unstructured data）

你有没有遇到过一个『某些东西的数组（array of stuff）』然后发现它并不只是一个列表？你是否将数组的键作为字段使用？你是否感受过不知道数组中到底有什么的痛苦？不确定其中的数据是否真的是你所期望的那样，或者哪些字段是否存在？

看一下我说的是什么意思: 使用 Laravel 的请求。把这个例子看成是一个基本的 CRUD 操作，用于更新一个现有的客户。

```
function store(CustomerRequest $request, Customer $customer)
{
    $validated = $request->validated();

    $customer->name = $validated['name'];
    $customer->email = $validated['email'];

    // ...
}
```

你可能已经发现了问题：我们不知道 `$validated` 数组中到底有什么数据。虽然在 PHP 中，数组是一种通用的、强大的数据结构，但只要它们被用来表示『事物列表』以外的东西，就有更好的方法来解决你的问题。

在给出解决方案之前，先来看看你可以通过什么方法来处理这种问题：

- 查看源码（Read the source code）
- 查看文档（Read the documentation）
- 打印出 `$validated` 的内容（Dump $validated to inspect it）
- 使用调试器查看（Or use a debugger to inspect it）

现在想象一下，你们团队好几个开发人员一起做一个项目，你的一个同事在五个月前写了这段代码。我可以向你保证，如果用上面的几个方法，你将不知道你在用的数据里有啥。

事实证明，强类型系统与静态分析相结合可以极大地帮助我们理解我们到底在处理什么。例如，像 Rust 这样的语言很好地解决了这个问题：

```
struct CustomerData {
    name: String,
    email: String,
    birth_date: Date,
}
```

我们需要这样的结构体，不行的是，PHP 没有结构体的概念，它只有数组和对象。

Emmm...对象和类也够用了。

```
class CustomerData
{
    public string $name;
    public string $email;
    public Carbon $birth_date;
}
```

它有点冗长，但做的事情差不多。这个简单的对象可以像这样使用。

```
function store(CustomerRequest $request, Customer $customer)
{
    $validated = CustomerData::fromRequest($request);

    $customer->name = $validated->name;
    $customer->email = $validated->email;
    $customer->birth_date = $validated->birth_date;

    // ...
}
```

这样，IDE 内置的静态分析器就能够告诉我们正在处理什么数据。

这种将非结构化数据包装成类型，使我们能够以一种可靠的方式使用这些数据的模式, 被称为『数据传输对象（data transfer objects）』。这是我强烈建议你在比一般的Laravel项目中使用的第一个具体模式。这是我强烈推荐你在大型 Laravel 项目中使用的第一种设计模式。

当你和同事、朋友或者 Laravel 社区讨论这本书的时候，你可能会发现对强类型持有不同看法的人。事实上有许多人更喜欢 PHP 的动态和弱类型特性，对于这点肯定是要拿出来说的。

根据我的经验，当与多个成员画大量时间开发项目的时候，强类型有更多的优势。你必须利用一切机会来减少认知负荷（cognitive load）。你不希望开发者在每次想知道某个变量的具体内容时都要调试他们的代码。这些信息必须明显的体现，这样开发人员就可以专注于重要的事情：构建应用程序。

当然，使用 `DTO` 是有代价的：不仅有定义这些类的开销；你还需要把数据映射到 `DTO` 上，例如请求数据。但是，使用 `DTO` 的收益绝对超过了这些额外的成本：无论编写 `DTO` 花了多少时间，从长远来看，都是能弥补回来的。

不过，关于从『外部』数据构建 DTO 的问题仍然需要解答。

## DTO 工厂（DTO factories）

我将分享两种构建 DTO 的方法，并说明我偏爱哪一种。

第一种是最正确的一种，使用专门的工厂类：

```
class CustomerDataFactory
{
    public function fromRequest(
        CustomerRequest $request
    ): CustomerData {
        return new CustomerData([
            'name' => $request->get('name'),
            'email' => $request->get('email'),
            'birth_date' => Carbon::make(
                $request->get('birth_date')
            ),
        ]);
    }
}
```

拥有一个独立的工厂可以使你的代码在整个项目中保持干净。我认为这个工厂在应用层是最合理的，因为它必须知道具体的请求和其他类型的用户输入。

虽然是正确的解决方案，但你是否注意到我在之前的例子中使用了一个快速用法？没错，在 DTO 类本身构建：`CustomerData::fromRequest()`。

这种方法有什么问题？一方面，它在『领域层（Domain）』中添加了特定于应用层的逻辑。位于『领域层（Domain）』的 DTO 类现在必须知道位于『应用层（Application）』中的 `CustomerRequest` 类。

```
use Spatie\DataTransferObject\DataTransferObject;

class CustomerData extends DataTransferObject
{
    // ...

    public static function fromRequest(
        CustomerRequest $request
    ): self {
        return new self([
            'name' => $request->get('name'),
            'email' => $request->get('email'),
            'birth_date' => Carbon::make(
                $request->get('birth_date')
            ),
        ]);
    }
}
```

显然，在领域层混合应用层的代码并不是个好主意。但是我更倾向于这种。这有两个原因。

首先，我们已经确定，DTO 是数据进入代码的入口。一旦我们从外部处理数据，我们就想把它转换成 DTO。我们需要在某个地方进行这种映射，所以我们不妨在它自身类中进行映射。

其次，这是更重要的原因；我更喜欢这种方法，是因为 PHP 自身的一个局限性：它不支持命名参数 —— 反正暂时还没有。

看，你不希望你的 DTO 最终有一个构造函数，每个属性都定义一个单独的参数：这不科学，而且在处理可空值（nullable）或默认值（default-value）时会比较难搞。这就是为什么我更喜欢将一个数组传递给 DTO，并让它根据数组中的数据来构造自己。顺便说一句：我们使用 `spatie/data-transfer-object` 包来做这件事。

由于不支持命名参数，也没有静态分析，这意味着你在构建 DTO 的时候，对需要什么数据一无所知。我更喜欢把这种『黑盒（being in the dark）』保持在 DTO 类中，这样在外部无需考虑太多就可以调用。

 如果 PHP 支持命名参数（将会在 `PHP 8` 中支持），我认为工厂模式是可行的：

 ```
public function fromRequest(
    CustomerRequest $request
    ): CustomerData {
        return new CustomerData(
            name: $request->get('name'),
            email: $request->get('email'),
            birth_date: Carbon::make(
                $request->get('birth_date')
            ),
        );
}
 ```

在 PHP 支持命名参数之前，我会更实在地使用第二种方法，而不是理论上正确的第一种。当然，这取决于你。自由地选择最适合你团队的方式。

## 类型属性的方案（An alternative to typed properties）

有一个使用类型属性的方案：`DocBlocks`。我前面提到的 DTO 包也支持。

```
use Spatie\DataTransferObject\DataTransferObject;

class CustomerData extends DataTransferObject
{
    /** @var string */
    public $name;

    /** @var string */
    public $email;

    /** @var \Carbon\Carbon */
    public $birth_date;
}
```

在某些情况下，DocBlocks 具有优势：它们支持类型数组和泛型。但在默认情况下，DocBlocks 并不能保证数据是他们所定义的那种类型。幸运的是，PHP 反射 API，有了它，就有了更多的可能性。

这个包所提供的解决方案可以被认为是 PHP 类型系统的扩展。虽然在用户领域和运行时能做的只有这么多，但它仍然是有价值的。如果你不能使用 `PHP 7.4`，而又想让你的 `DocBlock` 类型更加严谨，那么这个包可以满足你的要求。

## 关于 PHP8 中 DTO 的说明（A note on DTO's in PHP 8）

`PHP 8` 将支持命名参数，以及构造函数属性的提升。这两个特性将对你需要编写的重复代码量产生巨大的影响。

以下是 `PHP7.4` 中一个小型 DTO 类的样子。

```
class CustomerData extends DataTransferObject
{
    public string $name;

    public string $email;

    public Carbon $birth_date;

    public static function fromRequest(
        CustomerRequest $request
    ): self {
        return new self([
            'name' => $request->get('name'),
            'email' => $request->get('email'),
            'birth_date' => Carbon::make(
                $request->get('birth_date')
            ),
        ]);
    }
}

$data = CustomerData::fromRequest($customerRequest);
```

这是在 `PHP 8` 中的写法：

```
class CustomerData
{
    public function __construct(
        public string $name,
        public string $email,
        public Carbon $birth_date,
    ) {}
}

$data = new CustomerData(...$customerRequest->validated());
```

因为数据几乎是每个项目的核心，它是最重要的构建模块之一。数据传输对象（Data transfer objects）为你提供了一种以结构化、类型安全和可预测的方式来处理数据的方法。

你会注意到在本书中，DTO 的使用频率较高。这就是为什么一开始就对它们进行深入研究是非常重要的。同样地，还有一个关键的构建模块需要我们关注：动作（actions）。这是下一章的主题。
